# Test report: 1-on-1 mentoring request (request-then-approve, no booking/payment)

## Tests added
- `src/lib/mentoring.test.mjs` (new, 8 tests): DB-direct tests against a real Postgres, following the established pattern in `membership.test.mjs`/`account-dashboard.test.mjs`. Confirmed empirically that neither `src/app/mentoring/actions.ts` nor `src/app/admin/mentoring/actions.ts` can be imported into a plain Node test process — both import `src/lib/email.ts`, which has a top-level `import "server-only"` guard that throws unconditionally outside Next's own bundler (same constraint already documented for `src/lib/stripe.ts`/`src/lib/db.ts` in the prior two test files). Each test instead performs the exact same Prisma operation/query shape the real action's source uses for that step.

## Coverage of spec edge cases
- **Submitting a request for an inactive or deleted mentor**: covered — "the exact query createMentorRequest uses to validate a mentor correctly signals inactive vs missing" confirms both the `!mentor` (deleted/nonexistent id) and `!mentor.active` (deactivated) branches of the guard would evaluate true against real data, i.e. the guard's precondition data shape is correct. The guard's own control-flow (the actual `if` statement inside the action) isn't exercised directly, for the reason above — verified by reading the code instead.
- **Admin approves/declines a request twice**: partially covered — "MentorRequest status can be independently set to 'approved' or 'declined'" confirms that once a request transitions out of `"pending"`, it stays out of `"pending"` (the data precondition the idempotency guard checks), but the guard's own `if (request.status !== "pending") return;` branch can't be invoked directly for the same import-constraint reason. Verified by reading `src/app/admin/mentoring/actions.ts` directly: both `approveMentorRequest` and `declineMentorRequest` check this before doing anything else.
- **Resend not configured**: not covered by an automated test — same as the membership feature's equivalent edge case, this requires invoking the real action (blocked by the `server-only` constraint) or mocking network/module state, neither of which fits this repo's existing test harness. Verified by reading the code: every email send in both new action files is wrapped in `isResendConfigured()` with a `console.error` fallback, mirroring `forgot-password/actions.ts`'s exact established pattern.
- **No active mentors exist**: covered — "public mentor listing query only returns active mentors" confirms the exact query `src/app/mentoring/page.tsx` uses excludes inactive mentors; by extension an all-inactive-or-empty table naturally produces the empty array the page's `mentors.length === 0` branch checks for.
- **Admin deletes a Mentor with existing requests (`onDelete: SetNull`)**: covered — dedicated test creates a mentor + request, deletes the mentor, and confirms the request row survives with `mentorId: null` and its other fields untouched. This is the single most valuable test in the suite, since it exercises the actual migration's DDL/FK behavior, not just application code.
- **Admin deletes a Mentor with no requests**: covered — explicit test confirms this never throws (relevant since the FK is nullable/SetNull, not Restrict — a delete must always be allowed).
- **Long free-text `message`/`preferredTimes`, no length cap**: covered — 5000-character strings round-trip through both fields intact.
- Not in the original spec's edge case list but added since it's foundational to everything else in this feature: **new `Mentor` rows default correctly** (`active:true`, `notifyMentorOnRequest:false`, `sessionRateCents:null`) and **`MentorRequest` creation defaults `status` to `"pending"`**.

- `npm test` (full suite, `DATABASE_URL` set to a real local Postgres): **32/32 pass, 0 fail** (24 pre-existing + 8 new). No regressions in any other test file.
