# Test report: AI chat assistant for paid members

## Tests added
- `src/lib/chat.test.mjs` (new, 10 tests): DB-direct tests against a real Postgres, following the established pattern in `membership.test.mjs`/`mentoring.test.mjs`. Confirmed empirically that `src/app/account/chat-actions.ts` cannot be imported into a plain Node test process — it transitively imports `src/lib/dal.ts`, `src/lib/stripe.ts`, and `src/lib/openai.ts`, all `server-only`-guarded (same constraint documented in every prior test file this session). Each test performs the exact same Prisma operation/query shape the real code uses for that step.

## Coverage of spec edge cases
- **OpenAI not configured**: not covered by an automated test — exercising this requires calling the real action (blocked by the import constraint) or mocking module state, neither fitting this repo's existing test harness. Verified by reading `chat-actions.ts` directly: `isOpenAIConfigured()` is checked and returns a clear error before any OpenAI call is attempted, mirroring every other optional-integration check in this codebase.
- **0 free questions and 0 credits**: covered indirectly — "free-vs-credit determination matches dailyFreeQuestions boundary exactly" confirms the exact `isFree` boundary math; the credit-balance branch itself (`user.chatCreditBalance <= 0`) is a trivial comparison verified by direct code reading, not worth a separate DB test since it doesn't touch the database differently than the decrement test already covers.
- **Hard daily cap hit regardless of credits**: covered — "today's usage count query only counts ChatUsage rows from today (UTC)" confirms the exact count the hard-cap check compares against `settings.hardDailyCap`; the comparison itself (`questionsUsedToday >= settings.hardDailyCap`) is verified by direct code reading given it's evaluated before the free/credit branch entirely, so a true hard-cap hit never reaches the credit check either way.
- **Rate limiting**: covered — dedicated test confirms a `ChatUsage` row created "just now" falls inside the 3000ms window and one just past it doesn't, using the exact `Date.now() - lastUsage.createdAt.getTime() < RATE_LIMIT_MS` comparison from the real code.
- **Webhook for an unknown/unrelated `stripeSessionId`**: covered — confirms `findUnique` returns `null` rather than throwing, so the real code correctly falls through to the existing `Purchase`-based branch.
- **Duplicate webhook delivery for the same question-pack checkout session**: covered — dedicated test processes the same webhook event twice and confirms the credit is applied exactly once (`chatCreditBalance` stays at 10, not 20), and the purchase's `status` ends at `"paid"` either way.
- **Membership lapses mid-session (re-check on every request)**: covered — "a non-member... fails the exact eligibility check" confirms the boolean logic `sendChatMessage` evaluates on every call; since this check runs at the top of every request server-side (not cached from page load), this data-level test stands in for the "mid-session" scenario, which is really the same check evaluated at a later point in time.
- **OpenAI call fails/times out — must not consume a free question or credit**: not covered by an automated test — same constraint as the "not configured" case above (requires invoking the real action or mocking the OpenAI client). Verified by reading the code: the `try/catch` around the OpenAI call returns early on any thrown error or falsy reply, and the `db.chatUsage.create`/decrement calls are both unreachable from that early return — they only execute after a confirmed truthy `reply`.
- **ChatSettings row doesn't exist yet**: covered — dedicated test deletes the singleton row first, confirms the upsert creates it with schema defaults, then confirms a second upsert call does not reset a subsequently-modified row back to those defaults (the `update: {}` semantics that make this safe to call from every request without ever clobbering an admin's saved settings).
- Not in the original spec's edge case list but added since they're foundational to the rest: **`chatCreditBalance` defaults to 0** on new users, **`onDelete: Cascade` actually deletes** `ChatUsage`/`ChatPackPurchase` rows when their user is deleted (the real DDL behavior, not just the Prisma schema declaration — the single most valuable test in this suite, exercising the migration itself), and **the atomic `decrement` operator** behaves as expected.
