# Review: Quiz + Certificate feature

## Verdict
APPROVE

## Spec compliance
- **Schema**: `QuizSettings`, `QuizQuestion`, `QuizAttempt`, `QuizAttemptAnswer`, `Certificate` all match the spec exactly, including the deliberately-different `onDelete: SetNull` on `QuizAttemptAnswer.questionId` (vs. `Cascade` everywhere else in this diff) — verified as real DDL by the test suite deleting a `QuizQuestion` mid-attempt and confirming the answer row survives with `questionId: null` and is correctly excluded from grading, not just declared in the schema. `Certificate.userId @unique` + `Cascade` also verified as real DDL (duplicate-insert rejection test, cascade-on-user-delete test).
- **`maybeIssueCertificate`**: implements the exact required check order — existing-certificate short-circuit (this is what makes issuance permanent: nothing re-evaluates or revokes once a row exists) → membership → passing-attempt existence → "every published course with ≥1 lesson fully completed," computed fresh against the live published set each call. The short-circuit behavior is specifically test-verified against a user whose `membershipStatus` was changed to `"canceled"` after certificate issuance, directly demonstrating the "never revoked" business rule in code, not just in prose.
- **`setLessonCompletion`**: `maybeIssueCertificate` is called only in the `completed: true` branch, never on un-marking — matches the spec's stated reasoning exactly (un-marking can never newly satisfy eligibility).
- **`startQuizAttempt`/`submitQuizAttempt`**: check ordering matches the spec's required sequence (session/login → membership → rolling-window rate limit → question availability, for start; session/login → attempt ownership → already-submitted guard → server-locked grading, for submit). Grading correctly ignores any client-submitted `questionId` not present in the attempt's own server-created `QuizAttemptAnswer` rows — confirmed by a dedicated test that submits a fabricated third answer and verifies it's never counted. The correct answer (`correctOption`) is never sent to the client in `startQuizAttempt`'s response shape (`QuizQuestionForClient` omits it entirely) — verified by reading the type and the actual returned object literal.
- **Admin CRUD + pagination**: `admin/quiz-questions/` mirrors `admin/mentors/`'s file layout exactly; the `?page=N` pagination is a clean, correct query-param implementation (bounds-checked `Prev`/`Next` links, `Math.max`/`Math.ceil` guard against `page=0` or an out-of-range page crashing the query).
- **Certificate page**: confirmed to gate solely on `Certificate` row existence, no membership re-check anywhere in `src/app/account/certificate/page.tsx` — matches the "permanent" decision exactly. Uses `requireUser()` (real login required), correctly distinct from the course page's anonymous-token-preserving `verifyUserSession()` pattern, since there's no anonymous case to preserve here.
- **Print handling**: `.no-print` class + `@media print` rule + `window.print()` button is a minimal, dependency-free implementation exactly as scoped (no PDF library added).
- **Out-of-scope items honored**: no quiz question *content* anywhere in the diff, no bulk-import UI, `LessonCompletion`'s own logic untouched beyond the one new call site, no new PDF dependency in `package.json`.

## Issues found
- Non-blocking: `submitQuizAttempt` does not re-check membership eligibility at submission time (only `startQuizAttempt` does). A member whose subscription lapses in the few minutes between starting and submitting an attempt can still submit and potentially earn a certificate-qualifying pass. This exactly matches what the spec specified for this action (session/ownership/already-submitted checks only) — not a deviation from the spec — but worth flagging as a real, if narrow, business-logic gap for a future pass, parallel to `sendChatMessage`'s every-request re-check in the chat feature.
- Non-blocking: `startQuizAttempt`'s `[...publishedQuestions].sort(() => Math.random() - 0.5)` is the well-known "not a uniformly random shuffle" pattern (Fisher-Yates would be uniform; this biases certain permutations). For a quiz-question subset selection this bias is inconsequential — there's no fairness or security property riding on perfectly uniform shuffling here — so not worth blocking on, just noting for awareness.
- Non-blocking: same defense-in-depth note as every prior feature's review this session — no `select` narrowing on several `findMany`/`findUnique` calls (e.g. `db.quizQuestion.findMany({ where: { published: true } })` pulls `correctOption` and `explanation` server-side before filtering down to the client-safe shape in the return statement). This is correct and safe as written (the full row never reaches the client, only the explicitly mapped subset does), just flagging the pattern as it's been flagged before.

## Recommendation
Merge. Build, lint (0 errors; same 6 pre-existing warnings, none new), and the full test suite (63/63 pass, including all 12 new quiz/certificate tests, independently re-run rather than trusting the tester's report alone) all check out. The two properties most worth having gotten right — grading that can never be influenced by client-submitted data, and a certificate that's provably permanent once issued — are both implemented correctly and directly verified by tests exercising the real database behavior, not just the application code's intent.
