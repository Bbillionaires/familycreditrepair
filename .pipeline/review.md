# Review: My Account dashboard (linked Materials, Courses, Class Signups)

## Verdict
APPROVE

## Spec compliance
- `src/app/account/page.tsx`: read the complete `git diff`. All three queries (`materialPurchases`, `coursePurchases`, `signups`) use `{ equals: user.email, mode: "insensitive" }` exactly as the spec required — this was the single highest-risk detail in the whole feature (a plain equals would silently break for any historically-different-cased email), and I confirmed it directly in the code, not by trusting `changes.md`. ✅
- `status: "paid"` filter on both Purchase queries, matching the spec's confirmed-correct reading of `/api/download/[token]/route.ts` and the Stripe webhook as the sole "access granted" signal. `status: "pending"` rows are excluded by construction (not a separate filter step). ✅
- `dedupeByKey` helper matches the spec's exact code block; both call sites pass `materialId!`/`courseId!` as the key, and both are preceded by a `.filter((p) => p.material !== null)` / `.filter((p) => p.course !== null)` step, matching the spec's stated defense against a deleted Material/Course (whose `onDelete: SetNull` behavior already nulls the FK at the query level via `{ not: null }`, making this filter a second, intentional layer). ✅
- Upcoming/past signup split matches the spec's exact sort directions (upcoming ascending/soonest-first, past descending/most-recent-first) and rendering (past rows get `opacity-70`, matching the spec's "visually de-emphasized" instruction). ✅
- All three empty states present with the exact link targets the spec specified (`/materials`, `/courses`, `/calendar`). ✅
- Scope: no changes to `change-password-form.tsx`, `account/actions.ts`, or the logout form — exactly as the spec required staying untouched. No schema changes, no new dependencies. ✅
- Independently re-ran `npm run lint` and `npm test` (16/16 pass with a real database, clean skip-not-fail without one) myself — matches what coder/tester reported. ✅

## Issues found

- **Real but non-blocking: signup has no email-ownership verification, and this feature is the first place that gap has a visible consequence.** Tracing the actual risk: `Purchase`/`Signup` rows were created anonymously (no login required) before this feature existed, keyed only by a self-reported email address. This dashboard now surfaces that email's purchase/attendance history the moment *anyone* creates an account using that exact email — and `signupAction` (unchanged by this diff, already shipped) never verifies the registrant actually controls the inbox at signup time, only at password-reset time. Concretely: if attacker Bob knows victim Alice's email (e.g., from a leaked receipt, or simply guessing a common address) and Alice hasn't registered an account yet, Bob can sign up with Alice's email and immediately see what materials/courses Alice unlocked and which classes she signed up for. **I judge this non-blocking for this diff specifically** for three reasons: (1) the data exposed is low-sensitivity — only material/course *titles* and class *titles+dates*, never `Signup.phone`/`familySize`/`notes` (confirmed by reading the JSX — those fields are never rendered) and never payment details; (2) the underlying gap (no email verification at signup) is pre-existing, shipped in the prior User Accounts task, and this feature didn't introduce it — it just gave an existing gap a first real consequence; (3) fixing it properly (a verify-your-email-before-first-login flow) is a materially larger feature requiring Resend to be treated as non-optional, which conflicts with this app's established "degrade gracefully without Resend" design principle used everywhere else. **Recommend opening a follow-up task** ("decide whether to require email verification at signup, now that /account surfaces pre-existing purchase/signup history") rather than blocking this merge on it — this is a product/security-posture decision for the site owner, not an implementation defect in this diff.
- **No other security issues found.** The `email` value used in every query comes from `db.user.findUnique({ where: { id: userId } })`, where `userId` is derived from the verified session (`requireUser()`) — never from a request parameter a visitor could manipulate — so there is no path for a logged-in user to query anyone else's email. `mode: "insensitive"` is a structured Prisma/Postgres filter, not raw SQL or a user-controlled regex — no injection or ReDoS surface. `User.email` is written lowercase-only at signup and is `@unique`, so two different accounts can never collide on case-different variants of the same address — ruling out the (different, worse) scenario of two legitimate users' dashboards colliding with each other.
- **Test coverage is well-targeted and independently re-verified.** The tester's own three new permanent DB tests (`account-dashboard.test.mjs`) lock in exactly the right thing — the case-insensitive match itself, including a negative-control assertion that a plain equals genuinely fails (ruling out a false-positive test) — and I re-ran the full suite myself rather than trusting the reported numbers. The tester's Playwright pass caught and fixed a real gap in the coder's own test (the coder's dedupe test only checked "one row survives," which could pass even if the *wrong* row survived; the tester's version specifically asserts the surviving token is the newest of three, ruling that out) and used a stronger DOM-scoped assertion for the upcoming/past sectioning than the coder's text-order check. Both are meaningful strengthenings, not busywork.

## Recommendation
Merge as-is (ready to ship via `scripts/ship.sh`). Open one non-blocking follow-up task: decide whether `/signup` needs an email-verification step now that `/account` surfaces pre-existing purchase/class-signup history tied to whatever email is entered — a product/security-posture call for the site owner, not a defect in this diff.
