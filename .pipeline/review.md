## Issues found

- **The one deliberate, correct deviation from the spec's literal code, already caught and fixed by coder+tester — verified independently, not just trusted.** The spec's `requireUser` snippet calls `await destroyUserSession()` before redirecting on a stale `sessionVersion`. Next.js forbids mutating cookies during a Server Component render (only Server Actions/Route Handlers may), and `requireUser` is invoked from `/account/page.tsx`'s render path. The coder discovered this the hard way (a real "server error occurred" page, not a clean redirect) and removed the cookie-clear, redirecting only. I did not just read the code and accept the claim — I independently reproduced the exact triggering condition (signed up a real user, bumped its `sessionVersion` directly in the database to simulate a password change elsewhere, then hit `/account` with the now-stale cookie) and confirmed the current code redirects cleanly to `/login` with no error page. **This is a correct fix, not a regression from spec** — the spec's snippet was a reasonable first draft that happened to violate a Next.js runtime constraint the spec's author couldn't have verified without running it; the stale cookie is harmless (every future request re-checks and overwrites it on next login). Non-blocking; nothing further to do.
- **Minor, non-blocking: login has a small user-enumeration timing signal.** `loginAction` returns immediately with the generic error when `db.user.findFirst` finds no matching row, without performing a dummy password comparison — so a request for a nonexistent identifier returns measurably faster than one for a real account (which pays the cost of a real `scrypt` verification). This could in principle let an attacker distinguish "no such account" from "wrong password" via timing. This is a pre-existing class of risk this codebase has never defended against anywhere (the admin login has the same shape), and the spec explicitly already accepts a related, larger enumeration surface by design (the lockout message itself reveals an account exists, which the spec calls out and accepts deliberately in its edge-cases section). Not worth blocking on for a v1 of a free credit-education site's account system; flagging for awareness, not action.
- **No blocking security issues found.** Password hashing is real (scrypt, not a placeholder), session secrets are separate from the admin system as required, lockout is DB-backed (not in-memory, correctly matching this app's serverless deployment constraint), reset tokens are single-use/overwritten/high-entropy (nanoid(32) ≈ 192 bits, Prisma-parameterized lookup, no injection surface), and none of the forgot-username/forgot-password/signup-duplicate paths reveal which identifiers are registered to an unauthenticated caller (independently re-verified the forgot-username response is byte-identical for a registered vs. unknown email, not just visually similar).
- **Test coverage is appropriately scoped and independently re-verified, not just trusted.** The tester correctly identified that `server-only` blocks unit-testing almost all of this feature's logic outside Next's bundler (consistent with this repo's established, previously-confirmed finding), added a real permanent DB-level test file (`user-schema.test.mjs`) for the one thing that *is* safely testable that way (schema-level uniqueness constraints — a real backstop behind the app-level duplicate check, which only protects against the common case, not concurrent signups), and used independently-written Playwright runs for everything else, including catching the exact multi-context-vs-shared-context pitfall that would have made the "other device" test meaningless if two `Page`s in one `BrowserContext` had been used instead of two separate contexts. I re-ran the full test suite myself in both the with-`DATABASE_URL` and without-`DATABASE_URL` states and got the same results reported.
- **Not independently re-verified this pass (low risk, noted for completeness):** the admin system (`requireAdmin`, `/admin/*`) was not touched by this diff and I did not re-run the admin login/CRUD flows to confirm no regression. The diff to shared files (`site-header.tsx`, `dal.ts`) is purely additive — no existing code path in either file was altered, only new code appended — so the risk of a silent admin-side regression is low, but this is an assumption based on diff shape, not a fresh end-to-end admin test run in this pass.

## Recommendation
Merge as-is (ready to ship via `scripts/ship.sh`). No follow-up tasks required specifically from this feature — the two items already tracked from prior tasks (#12 CI `DATABASE_URL`, #14 Purchase FK delete behavior) are unrelated and unaffected by this diff.
