# Review: Cloudflare Turnstile bot-protection on signup/login/forgot-username/forgot-password

## Verdict
APPROVE

## Spec compliance
- `src/lib/turnstile.ts`: `isTurnstileConfigured()`/`verifyTurnstileToken()` match the spec's signatures exactly. Read the file directly — confirmed the fail-closed-on-missing-token (`if (!token) return false`, no network call) and fail-open-on-service-failure (`!res.ok` and the `catch` block both return `true`, after `console.error`) asymmetry is implemented exactly as specified, not just described. ✅
- `.env.example`: both variables documented with the exact "set both together" warning the spec called for. ✅
- `src/components/turnstile-widget.tsx`: renders `null` when unconfigured (confirmed by direct read — the early return happens before any script/div JSX), otherwise renders Cloudflare's script + `data-sitekey` div exactly as specified. ✅
- All four form files (`signup-form.tsx`, `login-form.tsx`, `forgot-username-form.tsx`, `forgot-password-form.tsx`): `<TurnstileWidget />` placed immediately before the submit button in each, confirmed via `git diff`. Specifically confirmed for the two forgot-* forms that the widget is only in the form-rendering branch, not inside the `state?.success` early-return branch (there's no form to attach it to there) — read the full file for `forgot-username-form.tsx` to confirm this directly rather than trust the description. ✅
- All four `actions.ts` files: read the complete diff for all four. The check is inserted at the exact spot the spec required in every case — after existing input validation (`parsed.success` / `emailResult.success` / the identifier-and-password guard), before any database read or write. This matters: a request that fails Turnstile never costs a DB round-trip, exactly as intended. ✅
- Scope: no changes anywhere outside the 11 files the spec listed. `/account` and `/account/reset-password` correctly left untouched, matching the spec's explicit exclusion. ✅
- Independently re-ran `npm run lint` and `npm test` (13/13 pass with a real database) myself — both pass, matching what the coder/tester reported. ✅

## Issues found

- **No blocking issues.** The core security property — a bot that omits the token field entirely gets rejected once Turnstile is configured, with zero exceptions — is implemented correctly and was independently verified by the tester against the real Cloudflare API using Cloudflare's own publicly-documented testing secrets, not mocked. I traced the same code path myself by reading `turnstile.ts` directly and cross-checking it against every call site.
- **Non-blocking, already transparently flagged by both coder and tester, and I agree with the assessment**: the real Turnstile widget script never successfully loaded in this sandbox's headless Chromium (`net::ERR_CONNECTION_RESET`), so the actual "user solves the widget → browser produces a real token → form submits it" path was never exercised in a live browser. What *was* verified, repeatedly and independently across all three agent passes: the server-rendered widget markup is correct, the server-side verification logic is correct against the real Cloudflare API (both the pass and fail testing secrets, plus a genuine unreachable-host failure for the fail-open path), and the missing-token/invalid-token rejections work end-to-end through the real running app. Since the client widget's only job is producing an opaque string and the server treats every token identically regardless of its origin, this residual gap doesn't undermine confidence in the feature's correctness — it's a sandbox limitation, not an application defect. Worth a one-time follow-up if this repo ever gets an environment with unrestricted egress for its headless browser, but not worth blocking this feature on.
- **Non-blocking observation, not a defect**: the tester's first configured-mode test run produced a false negative (signup succeeded when it should have been rejected) because the test server was restarted without `TURNSTILE_SECRET_KEY` in its runtime environment — the tester caught this themselves by inspecting `/proc/<pid>/environ` before trusting the result, diagnosed it as their own test-setup mistake rather than an application bug, and re-ran correctly. I re-traced this in the test report and agree with that read: `TURNSTILE_SECRET_KEY` is read live via `process.env` on every request (it's `"server-only"`, never bundled to the client), so it must be present in the actual serverless function's runtime environment in production — this is worth the site owner double-checking when they add the real key to Vercel (set on the Production/Preview environments the function actually runs under, not just locally), but that's an operational note for deployment, not a code defect to fix here.
- **No new information-disclosure vector found**: rejecting a request with `"Verification failed. Please try again."` before the forgot-username/forgot-password database lookup does not weaken the existing no-enumeration property — the response is now determined solely by whether Turnstile passed, which has nothing to do with whether the submitted email exists, so an attacker learns nothing about account existence either way. Confirmed by reading the insertion order directly (Turnstile check happens before `db.user.findUnique` regardless of what email was submitted).
- **No new dependency risk**: no npm package added; the only new external call is a plain `fetch` to Cloudflare's own documented REST endpoint, and the only new third-party script is Cloudflare's own widget script, loaded only when the site owner opts in by setting a site key. Consistent with this app's existing pattern of optional third-party integrations (Stripe, Resend) that are inert until configured.

## Recommendation
Merge as-is (ready to ship via `scripts/ship.sh`). No new follow-up tasks required — the one open item (verifying the real widget in a live browser) is explicitly non-blocking and already documented in `.pipeline/test-report.md` for whoever picks it up later, if ever.
