# Review: Course content type (structured, multi-lesson, free/paid)

## Verdict
APPROVE

## Spec compliance
- Schema: `Course`/`Lesson` models and the `Purchase.materialId`/`courseId` polymorphism match the spec exactly, migration generated against a real Postgres (not hand-written) as required. ✅
- Admin CRUD (courses + nested lessons), public listing/detail/success pages, Stripe checkout, and the lesson-file route all match the spec's file-by-file description point for point. Independently spot-checked the most security-relevant file (`src/app/api/courses/[id]/lessons/[lessonId]/file/route.ts`) line by line, not just trusting `changes.md`. ✅
- **Locked-view content-leak edge case**: independently re-verified this myself at the code level (confirmed `course.lessons[i].content`/`videoUrl`/`fileUrl` are never referenced anywhere in the `!hasAccess` JSX branch — only `lesson.title`) *and* by re-running the tester's own HTTP-level check description against the actual route code to confirm it matches. This is correctly implemented, and — importantly — it's not just "hidden via CSS": Next.js Server Components only serialize fields actually referenced in the returned JSX to the client, so unrendered fields fetched from the DB genuinely never reach the response. ✅
- **Flat-404, no-information-disclosure edge case** on the lesson-file route: verified directly — both the "lesson/file doesn't exist" branch and the "wrong token/course/status" branch return the identical `{error: "File not found or not available"}, 404` shape. No distinguishing signal. ✅
- Required fix to `/api/download/[token]/route.ts` (null-check for now-optional `purchase.material`) is correct, minimal, and independently re-confirmed via `npm run build`'s TypeScript check passing. ✅
- Independently re-ran `npm run lint`, `npm test` (9/9 pass), and `npm run build` myself rather than trusting the reports — all pass. ✅

## Issues found

- **`purchase.materialId`/`courseId` `onDelete` behavior changed from `RESTRICT` to `SET NULL`** (flagged by both coder and tester): confirmed via direct schema inspection — this is a real, automatic Prisma consequence of making the relation optional, not a coder oversight. **Non-blocking for this feature**, but worth a deliberate decision rather than leaving as an implicit side effect: before this change, deleting a Material/Course with existing purchases threw an unhandled error (already a rough edge — `deleteMaterial`/`deleteCourse` have no try/catch); after this change, deletion silently succeeds and orphans the purchase record's link to what was bought, with no admin-facing warning either way. For an app that handles real payments, I'd lean toward explicitly restoring `onDelete: Restrict` on both relations (protect purchase-history referential integrity, force an intentional decision before destroying that link) over silent orphaning — but this is a product/data-integrity judgment call, not a security bug, and there is currently no admin UI that would even display the orphaned link either way, so the practical blast radius today is low. **Recommendation: track as its own follow-up task, do not block this PR on it.**
- **Minor, non-blocking code cleanliness**: the lesson-file route's URL path includes a course `id` segment (`/api/courses/[id]/lessons/[lessonId]/file`) that is destructured out of `ctx.params` but never used or validated against anything — real authorization is correctly done via `lesson.courseId` read fresh from the database (compared against the purchase's `courseId`), not from the client-supplied path segment, so this is **not a security gap** (an attacker supplying a mismatched or garbage `id` segment gains nothing — access is still gated correctly). It's just a dead parameter that could confuse a future maintainer into thinking it's part of the authorization check when it isn't. Non-blocking; worth a one-line cleanup whenever this file is next touched, not urgent enough for its own task.
- **Test-report claim independently re-verified, not just trusted**: the DB-backed test suite's CI-skip behavior is the right call — confirmed by directly re-running `npm test` both with and without `DATABASE_URL` set (documented in test-report.md, and I re-ran both cases myself rather than taking that at face value): 9/9 pass with a real database, 6/9 pass + 3 clean skips (not failures) without one. This correctly avoids compounding the pre-existing, separately-tracked CI gap (task #12) while still providing real, meaningful local/future-CI coverage the moment that gap is fixed — no changes needed to the test file itself when it is.
- **No other security issues found** in the new file-serving route or access-token design: token lookups are exact-match against a unique-indexed nanoid(32) column (parameterized by Prisma, no injection surface, and brute-forcing 192 bits of entropy is infeasible regardless of any timing characteristics); `lesson.fileUrl` is only ever admin-supplied (same trust boundary as the pre-existing Material pattern); rendered lesson content goes through ordinary JSX text interpolation (React-escaped), not `dangerouslySetInnerHTML`, so admin-authored lesson content has no XSS vector even though it's longer-form text than other admin-authored fields elsewhere in the app.
- **Pre-existing pattern, not a new gap**: `requestFreeCourseAccess` has no rate limiting and no email-ownership verification, allowing unlimited free-access grants to arbitrary unverified email addresses. This is not new — it's identical to the already-shipped `requestFreeDownload` for Materials, and this app has no email-sending infrastructure at all yet (a known, separately-discussed future dependency). Not something to block this feature on, since it would be inconsistent to hold Course to a stricter standard than the Material pattern it deliberately mirrors.

## Recommendation
Merge as-is (ready to ship via `scripts/ship.sh`). Open two follow-up tasks, neither blocking: (1) a deliberate decision on `Purchase` FK delete behavior (`Restrict` vs. `SetNull`, possibly with an admin-facing warning either way) — this is a data-integrity/product decision, not a defect; (2) optionally drop the unused `id` path segment from the lesson-file route or add an explicit (redundant but self-documenting) check against it, purely for future-maintainer clarity, not because access control is actually broken without it.
